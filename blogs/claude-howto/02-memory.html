What Is Memory?

Memory in Claude Code comes in two forms:

1. Automatic Memory (Claude Web/Desktop)

Claude automatically synthesizes information from your conversations every 24 hours. You don't manage it; Claude remembers context about you, your projects, and your preferences.

2. Filesystem-Based Memory (Claude Code)

In Claude Code, memory is explicit: CLAUDE.md files that you create and version control. They live in your repository and define project standards, architecture, and preferences.

This post focuses on the second type ‚Äî filesystem-based memory in Claude Code.

The Power of CLAUDE.md Files

A CLAUDE.md file is like a README for Claude. It contains everything Claude needs to know about your project:

  • Coding standards
  • Architecture decisions
  • Team conventions
  • Common commands
  • Known issues and workarounds

The genius? It's version-controlled and shared with your team. Everyone's Claude assistant reads the same standards.

The Three Ways to Manage Memory

Method 1: /init - The Fast Start

The first time I discovered /init, I was shocked at how simple it was:

# In your project directory
/init

That's it. Claude analyzes your project and creates a complete CLAUDE.md file with:

  • Detected technology stack
  • Common patterns from your codebase
  • Suggested coding standards
  • Project-specific configurations

My experience: I ran /init in a Node.js project. In 30 seconds, Claude had created a comprehensive CLAUDE.md that detected:

  • Express.js with TypeScript
  • Jest for testing
  • ESLint configuration
  • Git workflow patterns

It wasn't perfect, but it gave me an 80% complete starting point that I could refine.

Method 2: /memory - The Editor Approach

When you need to make extensive updates, use /memory:

/memory

Claude opens your system's default editor with the memory file. You can:

  • Review existing memory content
  • Make bulk changes
  • Reorganize sections
  • Add detailed documentation

When I use it: Every few weeks, I run /memory to review and update project standards as our team's practices evolve.

Method 3: # - The Quick Add

This is my favorite for daily use. During any conversation, just start with #:

# Always use TypeScript strict mode in this project

Claude immediately asks which memory file to update, adds your rule, and continues the conversation.

Real example from last week:

Me: # Prefer async/await over promise chains

Claude: I'm saving this rule to memory. Which file?
        1. Project memory (./CLAUDE.md)
        2. Personal memory (~/.claude/CLAUDE.md)

Me: 1

Claude: ‚úÖ Memory saved! Now, about that API endpoint you asked about...

No interruption. No context switching. Just instant memory update.

Alternative patterns that work:

  • # new rule into memory: Always validate input with Zod
  • # remember this: Run tests before every commit
  • # add to memory: Use kebab-case for file names

My First CLAUDE.md: Project Memory

Here's the CLAUDE.md file I created (with help from /init) for an e-commerce project. It transformed how Claude worked with our codebase:

# Project Configuration

## Project Overview
- **Tech Stack**: Node.js, PostgreSQL, React 18, Docker
- **Team Size**: 5 developers
- **Coding Standard**: ESLint (airbnb config), Prettier

## Development Standards

### Code Style
- Use 2-space indentation
- Maximum line length: 100 characters
- Use async/await (not callbacks)
- Prefer functional components with hooks (not classes)

### Naming Conventions
- **Files**: kebab-case (user-controller.js)
- **Classes**: PascalCase (UserService)
- **Functions/Variables**: camelCase (getUserById)
- **Constants**: UPPER_SNAKE_CASE (API_BASE_URL)
- **Database Tables**: snake_case (user_accounts)

### Git Workflow
- Branch names: `feature/description` or `fix/description`
- Commit messages: Follow conventional commits
- PR required before merge
- Minimum 1 approval required

### Testing Requirements
- Minimum 80% code coverage
- All critical paths must have tests
- Use Jest for unit tests
- Use Cypress for E2E tests

### API Standards
- RESTful endpoints only
- JSON request/response
- Version endpoints: `/api/v1/`
- Document all endpoints with examples

### Database
- Use migrations for schema changes
- Never hardcode credentials
- Use connection pooling
- Enable query logging in development

## Common Commands

| Command | Purpose |
|---------|---------|
| `npm run dev` | Start development server |
| `npm test` | Run test suite |
| `npm run lint` | Check code style |
| `npm run build` | Build for production |

## Known Issues & Workarounds
- Safari 14 compatibility issues with async generators
- Workaround: Use Babel transpiler

The result? Claude immediately started:

  • Using correct indentation
  • Following naming conventions
  • Suggesting appropriate testing approaches
  • Respecting our API standards

No more repeating myself. Ever.

Directory-Specific Memory: Module Rules

As projects grew, I discovered the power of directory-specific memory. Here's an example for our API module:

File: ./src/api/CLAUDE.md

# API Module Standards

This overrides root CLAUDE.md for everything in /src/api/

## Request Validation
- Use Zod for schema validation
- Always validate input
- Return 400 with field-level validation errors

## Response Format

All responses follow this structure:

```json
{
  "success": true,
  "data": { /* actual data */ },
  "timestamp": "2025-11-08T10:30:00Z",
  "version": "1.0"
}

Error responses:

{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "User-friendly message",
    "details": { /* field errors */ }
  },
  "timestamp": "2025-11-08T10:30:00Z"
}

Pagination

  • Use cursor-based pagination (not offset)
  • Include hasMore boolean
  • Limit max page size to 100
  • Default page size: 20

Rate Limiting

  • 1000 requests/hour for authenticated users
  • 100 requests/hour for public endpoints
  • Return 429 when exceeded
Now, when working in the `/src/api/` directory, Claude automatically applies these API-specific rules without me saying a word.

## Personal Memory: My Preferences {#personal-memory-my-preferences }

Personal memory (`~/.claude/CLAUDE.md`) stores preferences that follow you across all projects:

```markdown
# My Development Preferences

## About Me
- **Preferred Languages**: JavaScript, Python
- **Communication Style**: Direct, with examples
- **Learning Style**: Show code first, explain after

## Code Preferences

### Error Handling
I prefer explicit try-catch blocks with meaningful error messages.
Always log errors for debugging.

### Comments
Use comments for WHY, not WHAT. Code should be self-documenting.

### Testing
I prefer TDD (test-driven development).
Write tests first, then implementation.

### Debugging
- Use console.log with prefix: `[DEBUG]`

- Show concrete examples before theory
- Include before/after code snippets
- Summarize key points at the end

This travels with you. Every project, every session, Claude remembers these preferences.

Importing External Documentation: The @ Syntax

Here's a powerful feature I discovered: you can import external documentation into memory using the @path/to/file syntax:

# Project Configuration

## Architecture
@docs/architecture.md
@docs/api-standards.md
@docs/database-schema.md

## Commands Reference
@package.json
@README.md

How it works:

  • Claude automatically includes the referenced file's content in its context
  • Supports both relative and absolute paths
  • Enables recursive nesting up to 5 levels deep
  • Avoids duplication across memory files

Real-world example:

Instead of copying your entire README into CLAUDE.md, just reference it:

# Project Overview
See @README.md for full project description and setup instructions

# Available Commands
See @package.json for all npm scripts

# Architecture Decisions
See @docs/adr/001-database-choice.md
See @docs/adr/002-auth-strategy.md

Why this is brilliant:

  1. Single source of truth: Update README once, Claude sees the changes
  2. No duplication: Keep your documentation DRY
  3. Team consistency: Everyone references the same docs
  4. Maintainable: As docs evolve, memory stays current

Import nesting example:

You can nest imports up to 5 levels:

CLAUDE.md

This changed how I structure project documentation. Now my CLAUDE.md is a roadmap to documentation, not the documentation itself.

How Memory Loads Automatically

One of the best parts? You don't have to do anything for memory to load.

When Claude Code starts:

  1. It traverses upward from your current directory
  2. Discovers all CLAUDE.md files in the hierarchy
  3. Loads them automatically with the correct precedence
  4. Merges them into a unified context

Example: If I'm working in /project/src/api/endpoints/:

Claude discovers and loads:
1. /Library/Application Support/ClaudeCode/CLAUDE.md (Enterprise)
2. /project/CLAUDE.md (Project)
3. /project/src/api/CLAUDE.md (API module)
4. ~/.claude/CLAUDE.md (Personal)

Result: All four contexts merged with proper precedence

No commands. No configuration. Just works.

Troubleshooting

Memory Not Being Applied

Problem: Claude doesn't follow CLAUDE.md rules

Solutions:

  • Verify file is named CLAUDE.md (case-sensitive)
  • Check file is in correct directory
  • Ensure Claude Code session loaded after file creation
  • Review memory file for syntax errors
  • Restart Claude Code session

Conflicting Memory Rules

Problem: Different memory files conflict

Solutions:

  • Remember hierarchy: Project > Directory > Personal
  • Check which memory takes precedence
  • Explicitly override in higher-priority file
  • Document intentional overrides

Memory File Too Large

Problem: Claude struggles with very large memory files

Solutions:

  • Trim to essential information only
  • Move detailed docs to external files (@import)
  • Split into directory-specific memories
  • Target 100-300 lines maximum

Memory Not Updating

Problem: Changes to CLAUDE.md don't take effect

Solutions:

  • Restart Claude Code session
  • Verify file was saved
  • Check for syntax errors in YAML/Markdown
  • Clear cache: rm -rf ~/.claude/cache
